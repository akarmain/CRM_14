# DOC-ARC-001 Бизнес-процесс и пользовательский сценарий

О документе: описывает основной бизнес-процесс CRM_14 — создание и обработку лида.

## 1. Основной процесс

Главный процесс CRM_14 — обработка лида от создания до завершения.

Цепочка стадий:

```text
new → qualified → proposal → won/lost
```

## 2. Сценарий создания лида

1. Пользователь открывает CRM_14.
2. Система показывает экран выбора роли.
3. Пользователь выбирает роль менеджера или РОП.
4. Пользователь нажимает «Добавить лид».
5. Система открывает форму создания лида.
6. Пользователь заполняет:
    - название;
    - комментарий / заметку;
    - источник.
7. Backend проверяет валидность данных.
8. Если данные корректны, лид сохраняется.
9. Система создаёт лид со стадией `new`.
10. Лид отображается в таблице и канбане.
11. Если данные некорректны, система показывает ошибку.

## 3. Сценарий движения лида

1. Менеджер открывает свой список лидов.
2. Выбирает лид в таблице или канбане.
3. Переводит лид на следующую стадию.
4. Система проверяет допустимость перехода.
5. Система закрывает предыдущую запись истории через `left_at`.
6. Система создаёт новую запись в `leads_stage`.
7. Система обновляет `leads.current_stage`.
8. История лида становится доступна в модальном окне.

## 4. Сценарий возврата стадии

1. Менеджер понимает, что стадия выбрана ошибочно.
2. Менеджер нажимает «Возврат».
3. Система создаёт заявку на возврат.
4. РОП открывает очередь возвратов.
5. РОП видит текущую и целевую стадию.
6. РОП одобряет или отклоняет возврат.
7. Если возврат одобрен, стадия изменяется.
8. Если возврат отклонён, заявка закрывается без изменения стадии.

Принятое допущение: в текущей схеме БД нет отдельной таблицы возвратов. В финальной версии нужно сверить реализацию и, если необходимо, описать возврат как временное состояние UI/backend.

## 5. Сценарий аналитики

1. Аналитик выбирает свою роль.
2. Открывает таблицу или канбан.
3. Применяет фильтры.
4. Смотрит распределение лидов по стадиям.
5. Анализирует конверсию.
6. Анализирует длительность стадий.
7. Экспортирует данные при необходимости.

## 6. Сценарий РОП

1. РОП выбирает роль.
2. Видит все лиды.
3. Фильтрует по менеджеру, источнику и периоду.
4. Проверяет воронку.
5. Открывает историю спорных лидов.
6. Одобряет или отклоняет возвраты.
7. Экспортирует отчёт.
