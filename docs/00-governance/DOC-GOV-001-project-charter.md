## 1. Контекст проекта

CRM_14 разрабатывается как учебный MVP мини-CRM. В рамках проекта команда моделирует работу отдела продаж, где лид проходит стадии от первичного обращения до выигрыша или потери.

Проект показывает, как можно организовать базовый цикл работы с лидами:

1. лид создаётся вручную или через импорт;
2. менеджер ведёт лида по стадиям;
3. система хранит историю стадий;
4. аналитик и руководитель видят воронку;
5. руководитель контролирует возвраты стадий;
6. данные можно экспортировать для проверки и отчётности.

## 2. Проблема

Компании могут терять лидов, если данные о них хранятся разрозненно: часть информации находится в таблицах, часть — в переписках, часть — у отдельных менеджеров. В такой ситуации сложно понять:

- сколько лидов сейчас в работе;
- на каких стадиях они находятся;
- какие источники приводят лиды;
- какие менеджеры отвечают за обработку;
- где лиды задерживаются;
- сколько лидов доходит до успешного закрытия.

Из-за этого падает прозрачность воронки, уменьшается управляемость процесса продаж и усложняется работа руководителя.

## 3. Цель проекта

Цель CRM_14 — разработать MVP CRM-системы, которая централизует работу с лидами, показывает воронку продаж и предоставляет базовую аналитику по эффективности процесса.

## 4. Бизнес-цель

Бизнес-цель проекта:

Повысить прозрачность обработки лидов и дать руководителю отдела продаж и аналитику инструмент для контроля конверсии, длительности стадий и качества работы с лидами.

## 5. Для кого делается система

Основные роли:

- менеджер продаж;
- аналитик;
- руководитель отдела продаж.

Система учебная, поэтому заказчиком считается учебная организация / вуз / преподавательский контур.

Принятое допущение: в финальной версии можно указать конкретную кафедру, дисциплину или преподавателя, если это требуется.

## 6. In scope

В MVP входят:

- выбор роли пользователя;
- создание лида;
- просмотр списка лидов;
- просмотр лидов в канбан-доске;
- изменение статуса лида;
- движение лида только по допустимой цепочке стадий;
- запрос возврата на предыдущую стадию;
- одобрение или отклонение возврата руководителем отдела продаж;
- просмотр истории стадий лида;
- комментарии к изменениям как дополнительная функция;
- фильтрация по источнику;
- фильтрация по менеджеру;
- фильтрация по периоду создания лида;
- импорт CSV/XLSX;
- экспорт Excel/CSV;
- расчёт конверсии;
- расчёт средней длительности стадий;
- отчёт по воронке продаж.

## 7. Out of scope

В проект не входят:

- промышленная авторизация и регистрация;
- реальные персональные данные клиентов;
- телефония;
- email-рассылки;
- интеграция с внешней CRM;
- промышленный деплой;
- полноценная ролевая модель безопасности;
- ML-прогнозирование продаж;
- расчёт финансовой суммы сделки;
- хранение телефона или email клиента;
- хранение коммерческой тайны или персональных данных.

## 8. Критерии успеха

Проект считается успешным, если:

- пользователь может выбрать роль и попасть в соответствующий интерфейс;
- менеджер видит только своих лидов;
- руководитель отдела продаж видит все лиды;
- аналитик видит данные для анализа, но не редактирует лиды;
- лид может быть создан вручную;
- лид может быть импортирован из CSV/XLSX;
- лид проходит стадии `new → qualified → proposal → won/lost`;
- возврат на предыдущую стадию требует одобрения РОП;
- история стадий сохраняется и отображается;
- воронка продаж показывает распределение лидов по стадиям;
- система считает конверсию и среднюю длительность стадии;
- данные можно экспортировать в Excel/CSV;
- тестовый набор содержит 100 лидов;
- документация описывает требования, БД, API, роли, тесты и демо.

## 9. Основные ограничения

- Проект выполняется в учебном формате.
- Данные являются тестовыми.
- Выбор роли заменяет полноценную авторизацию.
- PostgreSQL-схема используется как логическая модель.
- 1С используется в учебном контуре и связана с backend через HTTP.
- Время проекта ограничено.
- Часть дат в документации требует ручной корректировки.
